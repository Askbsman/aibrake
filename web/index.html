01Problem

AI agents burn money when they retry the same failed task without new evidence.

The 7th paid retry on the same TS2307 error rarely produces a different answer. The 4th opus call after a sonnet got stuck rarely teaches the model something it didn't already see. Most operators discover this in the monthly invoice — not before the call fires. Agent Spend Guard makes that decision before you pay.

02What it catches

Five deterministic detectors. No LLM-on-the-side. No prompt content read. Pattern matches return in milliseconds with versioned policy@ and detector@ stamps.

stale_context_retry_storm

Same failure fingerprint, growing retry count, no new files / tests / logs / diff between attempts. The canonical "7th guess" case.

same_tool_retry_loop

Same tool, same input, no diagnostic step between calls. Per-objective threshold (default 6, tunable via detector_policy).

model_escalation_without_evidence

Premium model burn — opus / GPT-5 / claude-4 — invoked on the same failure with no new evidence. Returns structured model_route if secondaryModel declared.

objective_drift

Next action falls outside objective.allowed_actions or hits objective.blocked_actions. Deterministic block.

task_budget_breach

Hard budget exceeded. The one detector that does block deterministically — your call whether to wire it as enforcing.

03How it works

Stateless Core. Stateful adapter on the agent side tracks retry history per objective. One POST, one decision.

Agent runtime
    │
    ▼  AgentActionTelemetry
CodingAgentAdapter
    │
    ▼  POST /v1/check  (Bearer asg_v1_…)
Agent Spend Guard
    │
    ▼  decision + recommended_policy + suggested_action (+ model_route)
SDK helper
    │
    ▼  guard.checkOrConfirm / checkOrDowngrade / checkShadow
Agent decides

The SDK contract has four helpers. checkShadow never blocks — it logs the decision and returns it. Recommended for the first 7 days of integration.

04Integrate in 5 lines

Drop-in middleware. No new SDKs to learn, no DB, no dashboard, no SaaS account. Just an HTTP POST per expensive action.

import { SpendingGuard } from "spending-guard";

const guard = new SpendingGuard({
  baseUrl: "https://api.agentspendguard.com",
  apiKey: process.env.ASG_KEY,
  failureMode: "open",   // never takes your agent offline on guard outage
});

// Shadow mode for the first 7 days — log the decision, never block.
const result = await guard.checkShadow(input);
if (result.projected_savings) {
  log.info({
    saved_usd: result.projected_savings.amount_usd,
    basis: result.projected_savings.basis,
    pattern: result.pattern,
  }, "guard caught a wasteful retry");
}

09FAQ

Will the guard slow down my agent?

Median /v1/check latency is ~5 ms against a steady-state hosted instance (4-8 ms is the typical range). At 95th percentile under benchmark load it stays under 20 ms. The SDK default timeout is 500 ms; if the guard is unreachable you fail open by default — a synthetic allow result is returned and your agent keeps running.

What if the guard service is down?

The SDK defaults to failureMode: "open" — transport-class errors (DNS, timeout, 5xx) are converted to a synthetic allow with pattern: "guard_unavailable". Your agent never goes offline because of us. You can configure "closed" if you'd rather hard-stop on guard outage, or "throw" if you want to handle the error yourself.

Note: server-side 4xx (validation errors, bad keys) are NOT failed open — they propagate so you see your integration bug instead of silently getting allow. This is the Stage 0.4.1 / 0.4.2 contract.

What data do you log?

Hashes and counts only. No raw prompts, no source-file contents, no API keys, no test output bodies. The decision log line carries: request_id, input_hash (sha256-16 of canonical input), api_key_hash (sha256-16 of the bearer key — never the raw key), the partner-supplied objective_id / actor_runtime labels, the decision + pattern + matched rules, and the partner-supplied next_action.estimated_cost + computed projected_savings.

Full contract in SECURITY.md.

How is this different from PQS / Boundary Guard / x402station?

PQS scores one prompt at a time on an 8-dimension rubric — answer to "is this single prompt worth paying for?"

Boundary Guard validates outputs against schema / safety constraints — "did the output pass?"

x402station checks endpoint trust before payment — "is this x402 endpoint safe to call?"

Agent Spend Guard watches the session loop — "is the agent already stuck in a paid retry pattern?" Per-session history with universal evidence model. We compose with the others — a serious agent stack uses prompt-preflight → loop-detection → output-validation in sequence.

How does the guard decide what's "stale"?

The universal evidence model. Every action carries six annotations on the current attempt: filesRead, testsRun, logsRead, gitDiffChanged, toolResultsChanged, contextSourceConfirmed. Any one of them > 0 / true counts as "the agent gathered new evidence between attempts."

Read-edit-rerun on the same failing test = allow (Stage 0.5.1 contract). Blind retry with zero annotations = require_confirmation with projected_savings.basis = "projected_future_attempts".

Can I tune the thresholds?

Yes, per-request via objective.detector_policy. Four knobs: same_tool_retry_threshold (default 6), premium_retry_without_evidence_threshold (default 3), expensive_action_usd_threshold (default 0.10), require_confirmation_after_repeats (default 5). Type / default / min / recommended-range / description for each are exposed at GET /v1/meta.

For scraper agents costing $0.10+ per call, tighter thresholds make sense (e.g. same_tool_retry_threshold: 3). For coding agents at $0.02-0.05 per call, defaults are usually fine.

What does the projected_savings number mean exactly?

It's the dollar amount offered by the guard — what you save if you heed the recommendation. The guard can't know whether you actually did. Three computation bases:

• next_attempt_avoided — single attempt cost. Used when the decision is warn / req_confirm / block on a non-storm pattern.

• projected_future_attempts — cost × min(3, paid_attempts_on_same_failure). Used on stale_context_retry_storm to look forward at the loop's likely tail.

• model_downgrade_delta — primary_cost − secondary_cost. Used when a model_route.to is set. Falls back to a conservative 60% reduction when estimatedCostUsd on the secondary isn't declared (labeled as such in the explanation).

Do you support models I'm using? GPT? Gemini? Local?

Provider-agnostic. The guard never calls a model — it judges your next call. Send any provider / model name in next_action.provider / next_action.model and it works. The DEFAULT_DOWNGRADE_MAP at /v1/meta has heuristic entries for Anthropic / OpenAI premium tiers; for everything else declare your own model_policy.secondaryModel with estimatedCostUsd and the guard will route to it precisely.
